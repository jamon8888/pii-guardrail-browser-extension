## Problem

`CLAUDE.md` and `AGENTS.md` currently contain only the auto-generated GitNexus
code-intelligence block. There is no project-specific guidance: no
architecture overview, no build/test commands, no statement of the single
invariant that matters most to this project (the local-only privacy
boundary), and no skills covering the project's own multi-step workflows
(WASM build, NER model pipeline, release validation, benchmarking, UI
conventions). New sessions have to rediscover all of this by reading code and
docs from scratch every time.

## Goals

- Add project-specific guidance to `CLAUDE.md`/`AGENTS.md` above the existing
  GitNexus block, in the same terse MUST/NEVER house style, without touching
  the GitNexus block itself (it's regenerated by a hook after commits/merges).
- Add five new skills under `.claude/skills/privacy-guardrail/` covering
  workflows the GitNexus skills don't: WASM build, NER model pipeline,
  release validation, OpenPII benchmarking, Svelte/UI conventions.
- Add a project-level `.claude/settings.json` (checked into git, distinct
  from the existing `.claude/settings.local.json`) wiring three hooks that
  mechanically enforce the privacy boundary invariant instead of relying on
  the model remembering it every session.
- Keep `CLAUDE.md` and `AGENTS.md` byte-identical.

## Non-Goals

- No changes to the GitNexus block or GitNexus hooks (those are user-global
  and out of scope).
- No new CI workflows (`.github/workflows/` stays empty â€” out of scope for
  this change; CI setup would be a separate project).
- No symlinking `AGENTS.md` â†’ `CLAUDE.md`; both are kept as independent files
  manually synced (decided to keep behavior identical even on filesystems
  without symlink support).
- The hooks do not attempt to replace `npm run validate:ci` or full
  lint/svelte-check/test runs â€” those remain manual developer/CI discipline.
  The hooks exist solely to make the *privacy boundary* invariant
  mechanically enforced and to give fast advisory feedback on two known
  footguns (stale WASM after Rust edits, forbidden runtime primitives).

## Critical Invariant: Privacy Boundary

This is the one rule that must never regress. The project has no telemetry,
no analytics, no remote inference, and no automatic upload of clipboard
content, prompts, responses, detected entities, vault data, or feedback logs.

**MUST**
- Keep `src/` free of runtime network primitives (`fetch`, `XMLHttpRequest`,
  `sendBeacon`, `WebSocket`, `EventSource`, remote `importScripts`/`<script
  src="http...">`) except the one approved exception below.
- Keep Transformers.js configured local-only in
  `src/offscreen/ner-provider.ts`: `allowRemoteModels = false`,
  `allowLocalModels = true`, `localModelPath` via `chrome.runtime.getURL`,
  browser/filesystem caches disabled.
- Run `npm run check:privacy-boundary` before committing changes under
  `src/` or `package.json` dependencies (a hook does this automatically â€”
  see Hooks below).

**NEVER**
- Add a dependency matching analytics/telemetry/crash-reporting name
  patterns (sentry, posthog, segment, mixpanel, amplitude, datadog, bugsnag,
  rollbar, newrelic, google-analytics, or generic `analytics`/`telemetry`).
- Add a new runtime network primitive without documenting the user-visible
  reason, proving it cannot upload user content, and extending the
  allowlist in `scripts/check-privacy-boundary.js`.

The only currently approved exception is the `fetch(url, { method: 'HEAD'
})` local-asset existence probe in `src/offscreen/ner-provider.ts`.

See [`docs/release/privacy-boundary.md`](docs/release/privacy-boundary.md)
and the `release-validation` skill for full detail.

### 3. Hooks â€” project `.claude/settings.json`

New file at repo root `.claude/settings.json` (committed to git â€” distinct
from the existing untracked `.claude/settings.local.json`, which keeps
per-developer permission overrides).

Three hook scripts under `scripts/claude-hooks/` (Node `.cjs`, cross-platform,
no shell-syntax dependency):

**`scripts/claude-hooks/pre-commit-privacy-check.cjs`** â€” PreToolUse, matcher
`Bash`. Reads stdin JSON; no-ops (exit 0) unless `tool_input.command` matches
`/\bgit\s+commit\b/`. When matched, runs
`node scripts/check-privacy-boundary.js` (via `child_process.execSync`,
captured). On failure, writes the captured error list to stderr and exits
`2` â€” this blocks the `git commit` Bash call. On success, exits `0`.

**`scripts/claude-hooks/src-edit-privacy-scan.cjs`** â€” PostToolUse, matcher
`Edit|Write`. No-ops unless `tool_input.file_path` is under `src/`. Reuses
the already-exported `checkPrivacyBoundary(rootDir)` wholesale (same
function the blocking commit hook uses â€” DRY, no new exports needed since
`checkPrivacyBoundary` already runs the runtime-primitive scan, the
transformer-config check, and allowlist filtering together). If `errors` is
non-empty, prints them to stderr as a warning (advisory â€” always exits `0`,
never blocks). Re-scanning all of `src/` on each edit is acceptable: the
scan is pure regex/text matching over a few dozen files, not I/O- or
CPU-heavy.

**`scripts/claude-hooks/rust-edit-reminder.cjs`** â€” PostToolUse, matcher
`Edit|Write`. No-ops unless `tool_input.file_path` matches
`/crate[\\/]src[\\/].*\.rs$/`. Prints an advisory reminder to rerun
`npm run build:wasm` and exits `0`.

`.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node \"scripts/claude-hooks/pre-commit-privacy-check.cjs\"",
            "timeout": 30,
            "statusMessage": "Checking privacy boundary before commit..."
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "node \"scripts/claude-hooks/src-edit-privacy-scan.cjs\"",
            "timeout": 10
          },
          {
            "type": "command",
            "command": "node \"scripts/claude-hooks/rust-edit-reminder.cjs\"",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
```

All three scripts parse stdin defensively (malformed/missing JSON â†’ exit 0,
no-op) so they can never break an unrelated tool call.

## Testing

- `pre-commit-privacy-check.cjs`: unit test (or manual run) with a synthetic
  stdin payload for a `git commit` command, both with the repo in a passing
  and a deliberately-broken state (temporarily inject a `fetch(` call in a
  scratch file) to confirm block/allow behavior and exit codes.
- `src-edit-privacy-scan.cjs` / `rust-edit-reminder.cjs`: manual invocation
  with synthetic stdin payloads covering matching and non-matching paths.
- No changes to existing Jest/Rust test suites are required by this work.
